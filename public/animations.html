Horizon's Gate Modding — Animations

How to define, play, and reuse sprite animations through mod data files. This guide assumes you already know how to write a mod and load data into the game; it only covers the animation system.

Disclaimer

This article was written by an LLM by reading the decompiled code of Horizon's Gate version 1.6.04, not from official documentation or hands-on testing. It may contain mistakes — field behaviors, defaults, and edge cases especially — and details may differ in other versions. Treat it as a starting point and verify anything important against the game itself.

Where animations live

The base game's animations (punch, use, idle poses, weapon swings, and the default "" animation) are defined in compiled code, not in data files — so you won't find example [Animation] blocks shipped in SystemData. But the data pipeline fully supports authoring and overriding animations, and the base game's data references them everywhere (e.g. an action with casterAnimation=punch;). This guide reconstructs the system from the engine code so you can build your own.

How data is ingested

All mod data — animations included — flows through the same parser. A data file is a flat stream of bracketed type tags, each followed by key=value; pairs. Newlines are ignored; the semicolon terminates each field. The parser reads a tag, collects its fields into a dictionary, and hands them to the engine to construct the matching object.

Two type tags matter here:

  • [Animation] — creates (or clones) a named animation and stores it under its ID.
  • [Keyframe] — creates a single keyframe and appends it to the animation whose ID it names. Keyframes are matched to their animation purely by ID, so the [Animation] block must appear before its [Keyframe] blocks.
-- The shape of every animation definition
[Animation]
    ID=myAnim;        -- the name other data references
[Keyframe]
    ID=myAnim;        -- links this frame to myAnim
    frame=0;
    endTime=0;
[Keyframe]
    ID=myAnim;        -- next frame of the same animation
    frame=1;
    endTime=0.12;

Values are parsed by reflection against the object's fields, so the value syntax depends on the field's type:

Field typeSyntaxExample
float / intplain number (invariant culture — always . for decimals)endTime=0.25
booltrue / falseflip=true
stringraw textfx=slash
Vector2x,y (comma-separated)offset=0,-6

The Animation object

An animation is a container for keyframes plus some playback-wide settings. It holds two parallel sets of tracks:

  • The base track — a single ordered list of keyframes. This drives the animation's timing: its total duration, when the "attack" lands, when FX fire, and the whole-body offset/rotation. For simple (un-rigged) sprites, the base track's frame is also the sprite that's drawn.
  • Per-body-part tracks — an optional separate keyframe list per limb (e.g. armR, head). Used to animate the individual pieces of a rigged humanoid independently of the body as a whole.

Invariant

Every animation must have at least one base keyframe. The base track defines the timeline; body-part tracks ride along on that same clock. duration is set automatically to the endTime of the last base keyframe you append, so you normally never set duration by hand — you let the base track define it. Body-part keyframes do not extend duration.

The Keyframe object

A keyframe is a target pose reached at a moment in time. The most important idea: a keyframe says "by time endTime, this part should be at this offset / rotation, showing sprite frame." The engine tweens the offsets and rotations from the previous keyframe's pose to this one over the intervening time. The sprite frame itself is not tweened — it snaps to the new value the instant the keyframe begins.

  • frame — which cell of the sprite sheet to draw. Snaps (no tweening).
  • offset — a pixel offset in screen space, tweened. Moves the sprite around (a bob, a lunge, a recoil).
  • directionalOffset — like offset, but rotated by the actor's facing, so positive X is always "forward" regardless of which way the actor looks. Use this for "step toward where I'm facing."
  • targetOffsetMult — multiplied by the vector pointing at the action's target. Set this to lunge toward whatever you're attacking.
  • endTime — the cumulative time within the animation at which this pose is reached (see below).
  • fx — names a visual effect spawned when this keyframe begins (blood, sparks, etc.).

What "tween" means

To tween (short for "in-between") is to generate the intermediate values between two keyframes automatically, so motion is smooth instead of jumping from pose to pose. You only specify the endpoints — a keyframe at offset=0,0 and the next at offset=0,-6 — and the engine fills in every step in between (0,-1, 0,-2, …) based on how far time has progressed between them. It's a straight linear blend: at the halfway point in time, you're at the halfway value. (If you've met the term elsewhere, this is exactly interpolation — the same idea by its formal name.) This guide uses "tween" throughout: offsets and rotations are tweened; the sprite frame is not — it snaps.

How playback works

The animator keeps a running time that increases each tick (scaled by the playback speed). endTime is cumulative and absolute within the animation, not a per-frame duration. So a three-keyframe animation might use endTime of 0, 0.1, 0.3 — meaning the first pose is held at the start, the second is reached 0.1s in, and the third 0.3s in (and duration becomes 0.3).

  1. The first keyframe usually has endTime=0 — it's the starting pose, reached instantly.
  2. For each later keyframe, the engine linearly tweens offset, directionalOffset, facingRotOffset, and targetOffsetMult from the previous keyframe's values to this keyframe's values, across the gap (this.endTime − previous.endTime).
  3. The frame sprite snaps when the keyframe is entered.

A keyframe with endTime 0 is a snap, not a hold

If a keyframe's endTime equals the previous keyframe's endTime (a zero-length slice), there's nothing to tween over and the engine moves straight through it. To hold a pose, give the next keyframe a later endTime. To guarantee a keyframe is actually drawn even if the clock overshoots it in one tick, set holdForAtLeastOneFrame=true.

Reusing a frame number

You'll often see the same frame value repeated on several consecutive keyframes. That's deliberate, and it follows directly from the rule that offsets tween but the sprite snaps. Holding one sprite still while changing the offsets around it produces motion of a fixed pose. The common reasons:

  • Slide a pose around without redrawing it. A lunge is typically three keyframes all using the same frame: rest → pushed forward (big offset / targetOffsetMult) → back to rest. The character never changes sprite; only the offset moves, so it reads as a smooth thrust and recoil. If you incremented frame instead, you'd get a flipbook, not a glide.
  • Hold an image for a beat. Repeat the same frame with a later endTime to pause on a pose (e.g. wind-up before a swing).
  • Animate timing/FX without changing art. Because the base track also carries fx and the attack timing, you sometimes add keyframes that only exist to place an fx or to mark the attack moment, leaving frame unchanged.

The contrast: a classic flipbook animation (a flag waving, a torch flickering) increments frame on each keyframe and barely touches the offsets. Most combat animations mix both — some keyframes swap the sprite, others reuse it to glide.

A second, unrelated kind of "reuse" is reusing a keyframe index when editing a cloned animation — covered in Example 3.

Triggering an animation

Animations are referenced by name (their ID) from several places. The two you'll reach for most are actions and actor-type slots; beyond those, scripts, world triggers, and status effects can all start an animation too.

Actions

An [Action] plays an animation on its caster via casterAnimation:

[Action]
    ID=mySlash;
    casterAnimation=mySlash_anim;
    casterAnimationSpeed=1;     -- scales playback; duration is divided by this
    casterAnimationLoops=0;

Related action fields:

FieldMeaning
casterAnimationThe animation ID to play. The special values weapon / weapon2 resolve to the equipped weapon's own animation.
casterAnimationSpeedPlayback multiplier. The effective length is duration / casterAnimationSpeed, so values > 1 are faster.
casterAnimationLoopsNumber of extra repeats; the clip plays casterAnimationLoops + 1 times total (default 0 = plays once). The caster stays busy for duration × (loops + 1) / speed.
casterAnimationDependsOnWeaponHandAppends the wielding hand index (0 or 1) to the name — see Example 3.
casterAnimationDependsOnFacingRotAppends the actor's facing to the name — see Example 3.

Actor types

An [ActorType] has a slot for each situational animation; set any of them to an animation ID. The full list is in Appendix: Actor-Type Animation Slots.

These aren't just spawn-time configuration — the engine fires them automatically in response to gameplay events. You fill the slots; the engine decides when each plays:

  • taking damage → painAnim; dying → deadAnim
  • blocking or dodging an attack → blockAnim / dodgeAnim
  • the stun / frozen / drown / sleep states → their matching slots
  • moving and terrain interactions → walkAnim / climbAnim / jumpAnim
  • using an action → chargeAnim / castAnim / spellcastAnim / swingAnim
  • any non-looping animation finishing → falls back to the idle slot on its own

An actor also carries a runtime idleAnimOverride that, when set (typically from a script or trigger), replaces the looping idle without editing the [ActorType].

Animation keywords (shorthand for the slots)

Where you name an animation for a specific actor, you can use a keyword that resolves to that actor's matching [ActorType] slot instead of a literal ID. So casterAnimation=pain plays whatever the caster's painAnim is — not a global animation literally named pain. The keywords mirror the slot list and are enumerated in Appendix: Animation Keywords. Any string that isn't a keyword is used as a literal animation ID, so keywords and explicit IDs coexist freely.

Keyword resolution applies to an action's casterAnimation, Lua animate(), and an actor value's animationOnHarmful. It does not apply to the animate trigger effect (below), which takes a literal animation ID only.

Keywords don't mix with hand/facing suffixes

If an action pairs a keyword with casterAnimationDependsOnWeaponHand or casterAnimationDependsOnFacingRot, the suffix is tacked onto the name (e.g. pain → pain0 / pain90). Unless a matching variant actually exists, the lookup finds nothing and the animation silently fails to play. Use suffixes with your own per-hand/per-facing IDs, not with keywords.

Lua scripts

Behavior and cutscene scripts can drive an animation directly. LuaManager registers an animate function that exposes the full animator entry point, so scripts get control that the data-only paths don't — arbitrary looping, a non-zero start keyframe, and tween-from-previous:

-- animate(actor, animationName, speed, startKeyframe,
--         interruptIfSameA, tweenFromPrev, loops, resumeIdleOnFinish)
animate(self, "ignite")                       -- only actor + name are required
animate(self, "idle_breathe", 1, 0, true, false, -1)  -- loops = -1 → forever

Defaults: speed=1, startKeyframe=0, interruptIfSameA=false, tweenFromPrev=false, loops=-1 (infinite), resumeIdleOnFinish=true. This is the path AI scripts use to pose actors.

World triggers

A trigger effect can play an animation on an actor as part of a scripted or zone event. Attach a [TriggerEffect] to a zone trigger (matched by numID) or a [GlobalTriggerEffect] to a global trigger (matched by ID), with effectID=animate:

[GlobalTriggerEffect]
    ID=smithStartsWork;
    effectID=animate;
    sValue=hammer_swing;    -- the animation ID to play
    sValue2=npc_smith;       -- which actor (omit to use the trigger's own actor)
    fValue=0.5;              -- optional: lock the actor "busy" this many seconds (also clears stun)

Status effects (actor values)

An [ActorValue] — the definition behind a status or element — can name an animation the target plays when a harmful effect of that actor value lands on it, via animationOnHarmful:

[ActorValue]
    ID=searing;
    animationOnHarmful=flinch_burn;

It only fires if the target's currently playing animation has interruptibleByDamage=true (and the target isn't drowning) — the same flag listed in the Animation fields reference. This is the hook for custom "got hit by X" reactions.

Example 1 — A simple lunge

The "hello world" of combat animations: the attacker glides toward its target and snaps back. There's no rigging here — one base track, and the same sprite frame on every keyframe (we're moving a pose, not swapping art). targetOffsetMult does the work of aiming the lunge at whatever's being hit.

[Animation]
    ID=lunge_simple;
    applyFrameToAllParts=true;   -- every part uses the base-track frame

[Keyframe]
    ID=lunge_simple;
    frame=0;
    endTime=0;               -- resting pose, at time 0

[Keyframe]
    ID=lunge_simple;
    frame=0;               -- SAME frame — we glide, not flipbook
    targetOffsetMult=0.5,0.5; -- lunge halfway toward the target
    endTime=0.1;             -- reached 0.1s in (fast push-out)

[Keyframe]
    ID=lunge_simple;
    frame=0;
    targetOffsetMult=0,0;     -- settle back to rest
    endTime=0.3;             -- slower recoil — duration is now 0.3

-- Hook it up to an action:
[Action]
    ID=jab;
    casterAnimation=lunge_simple;

Read it as three poses on one clock: rest at 0s, fully lunged at 0.1s, back to rest at 0.3s.

Example 2 — Rigged limbs, FX & the hit

Now a humanoid sword swing. The body's lunge stays on the base track, while individual limbs get their own tracks via bodyPart= — you can have as many as you need, each an independent keyframe list. Here the sword arm (armR) winds up and chops, and the head (head) adds a small follow-through. We place a slash fx on the contact keyframe and use attackKeyframe so the damage lands exactly when the blade connects, rather than at the start of the animation.

[Animation]
    ID=sword_chop;
    attackKeyframe=2;          -- base-track index where the hit registers

-- BASE TRACK: whole-body motion + the attack timing
[Keyframe]
    ID=sword_chop;
    endTime=0;                 -- index 0: rest
[Keyframe]
    ID=sword_chop;
    directionalOffset=-2,0;    -- index 1: lean back (wind-up)
    endTime=0.15;
[Keyframe]
    ID=sword_chop;
    directionalOffset=4,0;     -- index 2: drive forward — THE HIT
    targetOffsetMult=0.3,0.3;
    fx=slash;                  -- spawn the slash effect on contact
    endTime=0.22;
[Keyframe]
    ID=sword_chop;
    directionalOffset=0,0;     -- index 3: recover
    endTime=0.45;

-- ARM TRACK: the sword arm swings on its own sub-clock
[Keyframe]
    ID=sword_chop;
    bodyPart=armR;
    rotOffset=-40;             -- raised behind the head
    endTime=0.15;
[Keyframe]
    ID=sword_chop;
    bodyPart=armR;
    rotOffset=70;              -- chopped down through the target
    depthMod=1;                -- bring the arm in front of the torso
    endTime=0.22;
[Keyframe]
    ID=sword_chop;
    bodyPart=armR;
    rotOffset=0;               -- back to neutral
    endTime=0.45;

-- HEAD TRACK: a second, independent limb track on the same clock
[Keyframe]
    ID=sword_chop;
    bodyPart=head;
    rotOffset=-8;             -- tips back during the wind-up
    endTime=0.15;
[Keyframe]
    ID=sword_chop;
    bodyPart=head;
    rotOffset=10;             -- snaps forward with the chop
    endTime=0.22;
[Keyframe]
    ID=sword_chop;
    bodyPart=head;
    rotOffset=0;               -- settles back
    endTime=0.45;

[Action]
    ID=chop;
    casterAnimation=sword_chop;

Many tracks, one timeline

The base track and every limb track (armR, head, …) share the same time. Lining up their endTime values (here, all three have a keyframe at 0.22) is how you sync the body's forward drive, the arm's downward chop, the head's snap, and the slash FX — all landing on the attack keyframe. Add as many limb tracks as the rig has parts; each advances on the shared clock but holds its own poses.

Three ways to rotate a part

A body-part keyframe has three rotation knobs. rotOffset (used above) spins the part's own sprite. facingRotOffset adds to the part's facing rotation. Setting facingRotOverride=true makes facingRotOffset set the facing rotation absolutely instead of adding to it — reach for it when a part must point a fixed direction regardless of which way the actor is facing.

attackKeyframe timing — engine bug

attackKeyframe reliably gates when the hit lands by keyframe index. It also feeds a secondary value — the pause after the attack before the next action or turn resolves in turn-based combat — but this part is buggy as of v1.6.04. That pause is computed and cached the instant attackKeyframe is assigned, and it is never recomputed when keyframes are added afterward. Because an [Animation] block is parsed before its [Keyframe] lines, setting attackKeyframe on the [Animation] line measures it against an empty keyframe list, so the pause comes out as 0 and the next action follows immediately.

Workaround: assign attackKeyframe after the keyframes exist, by re-declaring the animation as a clone of itself (cloneFrom, introduced in Example 3). The clone copies the keyframes before applying attackKeyframe, so the value is measured correctly:

-- after all keyframes are added:
[Animation]
    ID=sword_chop;
    cloneFrom=sword_chop;   -- keyframes now exist on the clone source
    attackKeyframe=2;       -- now measured against real keyframe times

Example 3 — Cloning, mirroring & facing

Three reuse techniques that production animations lean on heavily.

Clone and tweak with cloneFrom

Start from an existing animation and override just what differs — far less error-prone than retyping every keyframe. To edit a specific frame of the clone, target its slot with replaceIndex (overwrite) or build on it with cloneFromIndex (copy-then-tweak). This is the index reuse mentioned earlier: you're naming an existing keyframe's position to modify it in place.

[Animation]
    ID=sword_chop_fast;
    cloneFrom=sword_chop;     -- inherit all keyframes + settings

-- Overwrite the contact frame to hit harder/sooner:
[Keyframe]
    ID=sword_chop_fast;
    replaceIndex=2;           -- replace base keyframe #2 outright
    directionalOffset=6,0;
    fx=slash_big;
    endTime=0.18;

-- Or: keep frame #1's pose but just add an FX, via copy-then-tweak:
[Keyframe]
    ID=sword_chop_fast;
    cloneFromIndex=1;         -- copy existing keyframe #1...
    replaceIndex=1;           -- ...and write it back to slot #1
    fx=windup_dust;            -- the only change

Mirror with flipArms

The off-hand version of a swing is just the same animation with the arm and held-item tracks swapped, and flipArms does exactly that on a clone — no need to author a mirror by hand. Its most common use is to build the pair of variants that casterAnimationDependsOnWeaponHand selects between. That flag (see Triggering an animation) appends the hand index (0 or 1) directly to the name, so the variants must be named sword_chop0 and sword_chop1 — flip the first to make the second:

[Animation]
    ID=sword_chop0;          -- main-hand swing (clone of the Example 2 animation)
    cloneFrom=sword_chop;

[Animation]
    ID=sword_chop1;          -- off-hand: a mirror of the main-hand swing
    cloneFrom=sword_chop0;
    flipArms=true;            -- swaps armL↔armR and item0↔item1

The 0/1 suffix only matters when you want casterAnimationDependsOnWeaponHand to pick the variant automatically. If you reference an animation directly by name instead, you can call it anything you like.

Per-facing variants with casterAnimationDependsOnFacingRot

Characters face four directions, and a swing often needs to look different depending on which way the actor is turned. Rather than branch inside one animation, the engine picks a different animation by name based on facing. When an action sets casterAnimationDependsOnFacingRot=true, the engine snaps the actor's facing to the nearest 90° and appends that number (0, 90, 180, or 270) to casterAnimation, then looks up that name. So you author one animation per facing:

[Action]
    ID=cast;
    casterAnimation=cast;
    casterAnimationDependsOnFacingRot=true;
-- At play time the engine looks up cast0 / cast90 / cast180 / cast270:

[Animation]
    ID=cast0;     -- facing one cardinal direction
-- ... keyframes for cast0 ...
[Animation]
    ID=cast90;    -- facing the next, etc.
[Animation]
    ID=cast180;
[Animation]
    ID=cast270;

Composing the suffixes

If you also set casterAnimationDependsOnWeaponHand=true, the hand index is appended first, then the facing — so the lookups become e.g. cast0_90 (hand 0, facing 90°) and cast1_270. With both flags on, you'd author one animation for each hand × facing combination. flipArms pairs naturally with the per-hand variants, since the two hands are mirror images.

There's also a conditional cousin, appendToCasterAnimation (with an accompanying formula), which tacks on a suffix only when a condition holds — letting one action choose between animation variants based on game state. Use it when facing/hand aren't enough to pick the right clip.

Example 4 — Looping & idle animations

The examples so far have been one-shot combat animations. Ambient ones — idle, walk, and the like — instead loop continuously, which raises a new concern: when the actor needs to do something else, the engine shouldn't cut the loop off mid-motion, it should finish to a natural rest pose first. That's what canStopHere marks. When asked to stop, the engine keeps playing until the loop reaches the nearest canStopHere keyframe, then exits cleanly.

[Animation]
    ID=idle_breathe;
    applyFrameToAllParts=true;

[Keyframe]
    ID=idle_breathe;
    frame=0;
    endTime=0;
    canStopHere=true;       -- the rest pose — safe to halt here

[Keyframe]
    ID=idle_breathe;
    frame=0;
    offset=0,-1;            -- chest rises (same frame, glides up)
    endTime=0.8;

[Keyframe]
    ID=idle_breathe;
    frame=0;
    offset=0,0;             -- settles; loops back to the top
    endTime=1.6;

-- Use it as an actor's resting animation:
[ActorType]
    ID=townsperson;
    idleAnim=idle_breathe;

Because only the rest frame is marked canStopHere, an actor interrupted mid-breath glides back down to neutral before switching to its next animation, instead of snapping. Mark more keyframes canStopHere if several poses are acceptable exit points.

Looping an action animation

To repeat an action's animation — a flurry of blows, say — set casterAnimationLoops on the action rather than looping the animation itself (see the action fields for its exact repeat-count semantics).

Example 5 — Sprite & palette overrides

Sometimes an animation needs to draw from an entirely different sprite sheet — a transformation, a special pose that doesn't exist on the normal rig, a creature that briefly becomes something else. The spriteOverride family swaps the art for the duration of the animation, reverting automatically when it ends.

[Animation]
    ID=ignite;
    spriteOverride=hero_ablaze;   -- draw from a different sheet
    paletteOverride=fire;          -- recolor for the duration
    spriteWidthOverride=48;        -- that sheet uses larger cells...
    spriteHeightOverride=48;
    spriteOffset=0,-8;             -- ...so nudge it up to keep the feet planted
    applyFrameToAllParts=true;

[Keyframe]
    ID=ignite;
    frame=0;
    endTime=0;
[Keyframe]
    ID=ignite;
    frame=1;
    endTime=0.15;
[Keyframe]
    ID=ignite;
    frame=2;
    fx=fire_burst;
    endTime=0.3;

The frame numbers now index cells of hero_ablaze, not the actor's normal sheet. applyFrameToAllParts makes every body part advance to the base keyframe's frame together — instead of each limb following its own per-part track — so the whole rig steps through the override sheet's frames in lockstep. That's what you want when each frame of the override sheet is a complete, pre-drawn pose. The width/height overrides tell the engine the override sheet's cell size, and spriteOffset compensates for the difference in alignment.

Gotchas

  • Order matters. The [Animation] block must be parsed before its [Keyframe] blocks, and before any cloneFrom that references it.
  • Always include a base track. At least one base keyframe is required; it defines duration and the attack timeline. Body-part keyframes alone won't play.
  • Offsets tween, sprites snap. Reuse a frame to glide a pose; increment frame to flipbook. (See Reusing a frame number.)
  • endTime is cumulative. Values must increase along a track. A zero-length slice is skipped, not held.
  • Decimals use a dot. Numbers parse under invariant culture; never use a comma as a decimal separator (commas separate Vector2 components).
  • Speed divides duration. casterAnimationSpeed=2 makes the clip play in half the time.
  • Per-facing animations are separate objects. With casterAnimationDependsOnFacingRot, define every name0/90/180/270 variant or the lookup will fail for the missing facing.
  • Never play the empty ("") animation. The built-in blank animation is a single keyframe roughly 1000 seconds long, so an actor that lands on it looks softlocked indefinitely. (A misspelled name is safer — it's logged and ignored, leaving the current animation running — but anything that resolves to "" will hang.)

Appendix: Niche Details

Edge cases you can ignore until you need them.

Built-in combat-entry animations

When an actor draws its weapons on entering combat, the engine plays one of three fixed-name built-in animations depending on how many hands are armed:

Animation IDPlayed when
unsheathe01both hands are armed
unsheathe0only the main hand is armed
unsheathe1only the off hand is armed

These are referenced by literal name, so you can customize them the way you'd customize any built-in animation: define an [Animation] with the same ID (unsheathe01, or punch, use, …), which replaces the existing definition — for a given ID, the last one loaded wins. (This is replacing the whole animation, distinct from the per-actor animation overrides like idleAnimOverride.)

Keyword resolution fine print

A few keywords (see Animation keywords) don't map one-to-one to a single slot:

  • stun falls back to painAnim when the actor type's stunAnim is empty.
  • swing / spellcast (and their hand-suffixed swing0/swing1/spellcast0/spellcast1 forms) resolve to swingAnim / spellcastAnim; if that slot is empty, the keyword is left as a literal name instead. This is why these two keywords survive the weapon-hand suffix where most others don't.
  • jump resolves to a built-in jump_gate animation (instead of jumpAnim) for actors that have the gateleap support ability.

Per-actor-type animation namespacing (specialAnim_prefix)

An [ActorType] can set specialAnim_prefix. When a non-keyword name is requested for that actor, the engine first checks for an animation called <prefix>_<name> and uses it if it exists. So with specialAnim_prefix=goblin, a script asking for taunt plays goblin_taunt if you've defined it — letting one shared script resolve to actor-type-specific animations without hardcoding IDs.

Appendix: Actor-Type Animation Slots

The slots an [ActorType] exposes, each set to an animation ID the engine plays on the matching event (see Triggering an animation).

Gameplay-state slots:

idleAnim, walkAnim, jumpAnim, climbAnim, painAnim, dodgeAnim, blockAnim, chargeAnim, castAnim, spellcastAnim, swingAnim, sleepAnim, stunAnim, frozenAnim, drownAnim, criticalAnim, deadAnim, mountedForceAnim.

Encounter-negotiation reactions — played as the actor reacts when approached to fight, make peace, or be bribed. They follow the pattern anim_try{Combat|Peace|Bribe}_<mood>_<outcome> (mood: overconfident / confident / uneasy / scared; outcome: fight / peace / flee / bribe). Not every combination exists — only these are defined:

anim_tryCombat_overconfident_fight, anim_tryCombat_overconfident_peace, anim_tryCombat_confident_fight, anim_tryCombat_confident_peace, anim_tryCombat_uneasy_fight, anim_tryCombat_uneasy_flee, anim_tryCombat_scared_flee.

anim_tryPeace_overconfident_fight, anim_tryPeace_overconfident_peace, anim_tryPeace_confident_fight, anim_tryPeace_confident_peace, anim_tryPeace_uneasy_fight, anim_tryPeace_uneasy_peace, anim_tryPeace_scared_peace, anim_tryPeace_scared_flee.

anim_tryBribe_overconfident_fight, anim_tryBribe_overconfident_peace, anim_tryBribe_overconfident_bribe, anim_tryBribe_confident_peace, anim_tryBribe_confident_bribe, anim_tryBribe_uneasy_bribe, anim_tryBribe_scared_bribe.

Appendix: Body-Part Names

When routing a keyframe with bodyPart=, use the rig's part names. The commonly animated ones:

head · torso · legs · armL · armR · item0 (item/weapon in hand 0) · item1 (item/weapon in hand 1) · tail. Also present on the rig: face · shoulderL · shoulderR · back.

Appendix: Animation Keywords

Keywords you can pass instead of a literal animation ID; each resolves to the actor's matching [ActorType] slot (see Animation keywords for how and where they apply):

idle, walk/move, jump, climb, pain, dodge, block, charge, cast, spellcast, swing, sleep, stun, frozen, drown, critical, dead.
